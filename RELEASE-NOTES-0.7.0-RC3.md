# SG-AWG-Panel 0.7.0-RC3

Дата: 2026-07-14

## Главное исправление RC3

Исправлена выдача VPN-адресов клиентам, созданным на удалённой SG-Node через Controller.

Раньше Controller учитывал только собственную базу клиентов. Если на SG-Node уже существовал локальный peer с адресом `10.77.0.2/32`, Controller мог выдать тот же адрес другому клиенту. Туннель такого клиента мог установить handshake, один раз передать трафик, а затем перестать работать из-за неоднозначного обратного маршрута.

В RC3:

- Node Agent передаёт Controller фактические `AllowedIPs` всех peers работающего `awg0`;
- peers, созданные Controller, помечаются отдельно от локальных peers SG-Node;
- новый клиент получает адрес с учётом реального состояния Node, а не только базы Controller;
- старый конфликтующий клиент автоматически переносится на следующий свободный адрес перед синхронизацией;
- Node Agent повторно проверяет список и не применяет конфигурацию, если адрес занят локальным peer;
- локальные peers сохраняются и не удаляются при синхронизации клиентов Controller.

Пример корректного результата:

```text
Локальный клиент SG-Node:       10.77.0.2/32
Клиент, созданный Controller:   10.77.0.3/32
```

После автоматического изменения адреса необходимо скачать актуальную конфигурацию клиента из Controller. Старая конфигурация с прежним адресом больше не является актуальной.

## Новая установка на любую EC2

На Controller и на будущей SG-Node запускается один и тот же полный установщик:

```bash
sudo bash 0.7.0-RC3-INSTALL-SG-AWG-PANEL.run
```

Каждая EC2 сначала остаётся самостоятельной полной панелью. Agent уже подготовлен, но запускается только после явного подключения сервера к выбранному Controller.

## Установка из ZIP

```bash
unzip 0.7.0-RC3-AWG-Panel.zip
cd 0.7.0-RC3-AWG-Panel
sudo bash install.sh
```

## Обновление существующей панели

Один и тот же updater запускается на обеих установленных EC2:

```bash
sudo bash 0.7.0-RC3-UPDATE-SG-AWG-PANEL.run
```

Обновление сохраняет базу, клиентов, Cluster, Cascade и действующее подключение Agent.

Перед релизом рекомендуется живой повторный тест на двух новых EC2: Controller, SG-Node, два клиента на одной Node с разными адресами, обычный выход и Cascade.
## Полное удаление одной командой

```bash
curl -fsSL https://raw.githubusercontent.com/s-gor/sg-awg-panel/v0.7.0-RC3/uninstall.sh | sudo bash
```

Скрипт удаляет Controller, SG-Node Agent, клиентов, ключи, базу, резервные копии, AWG-интерфейсы, Cascade, nftables/policy routing, AmneziaWG, Nginx и Certbot. Также очищаются legacy-службы и служебные каталоги прежних сборок. После удаления требуется `sudo reboot`.


## Исправление синхронизации ключа SG-Node

- работающий `awg0` на аутентифицированной SG-Node теперь является источником истины для серверного публичного ключа;
- Controller принимает подтверждённый текущий ключ из результата Agent и обновляет runtime до выдачи клиентского профиля;
- команда подключения Node передаёт реальный `awg_runtime` уже при регистрации, что устраняет окно со старым ключом между enrollment и первым heartbeat;
- исправлена версия Agent в bootstrap-подключении: `0.7.0-RC3` вместо устаревшего `208`;
- тесты GitHub Actions изолированы от боевого каталога `/var/lib/sg-awg-panel`.


## Единая модель установки

- отдельного пользовательского установщика SG-Node в релизе больше нет;
- полный установщик заранее размещает Agent и systemd unit на каждой панели;
- без `/etc/sg-awg-node/agent.env` Agent не запускается и сервер остаётся самостоятельным;
- после команды из Cluster тот же сервер начинает работать как SG-Node;
- общий updater обновляет и веб-панель, и активный либо ожидающий Agent.
