# SG-AWG-Panel 0.7.0-RC5

RC5 — единый следующий кандидат после RC4. Он объединяет новую кластерную адресацию, проверяемые резервные копии, совместимый экспорт AmneziaVPN, исправленную установку из GitHub `main` и сокращённое SSH-меню.

## Controller и до 12 SG-Node

Каждый сервер получает постоянный непересекающийся `/24`:

```text
Controller   10.77.0.0/24
SG-Node 1    10.77.1.0/24
SG-Node 2    10.77.2.0/24
...
SG-Node 12   10.77.12.0/24
```

- `awg0` каждого сервера использует `.1/24`;
- обычным клиентам выдаются `.2–.254/32` внутри пула своего сервера;
- номер и сеть закрепляются за Node в базе и показываются в Cluster и Clients;
- удалённый номер не переиспользуется автоматически;
- при обновлении с RC4 Controller переносит свои записи клиентов, а Agent атомарно меняет `awg0`, локальные peers и управляемые сетевые правила Node;
- перед применением Agent проверяет номер, сеть, адрес интерфейса, UDP `585`, ключ работающего `awg0` и полный набор peers;
- при ошибке конфигурация Node откатывается.

Ключи клиентов при миграции сохраняются. Клиенту, у которого изменился `Address`, нужно заново скачать и переимпортировать конфигурацию.

## Совместимость с официальным AmneziaVPN

Обычный полный туннель теперь экспортируется точной строкой:

```ini
AllowedIPs = 0.0.0.0/0, ::/0
```

Это применяется одинаково к локальным клиентам Controller, клиентам SG-Node, `.conf`, QR-коду и персональной ссылке. Ручное редактирование каждого нового файла не требуется.

Если администратор задал собственные маршруты или исключения, панель сохраняет рассчитанный пользовательский список сетей и не подменяет его полной парой.

## Резервные копии

Веб-панель и SSH используют одну функцию создания, проверки и восстановления.

Копия содержит:

- SQLite-базу панели, Clients, ключи, Cluster, Cascade и Traffic Rules;
- фактический `awg0.conf`;
- `web.env`, постоянный секрет сессий и хеш пароля;
- подключение Node Agent и его локальное состояние;
- управляемые файлы Nginx, заглушку, DNS и состояние Traffic Rules;
- управляемые файлы HTTPS/Let’s Encrypt для текущего домена;
- `metadata.json` и отдельный `manifest.json`.

Проверяются:

- отсутствие symlink/path traversal и разрешённые пути восстановления;
- SHA-256 и размер каждого файла, включая `metadata.json`;
- совместимость версии;
- `PRAGMA integrity_check` и обязательные таблицы SQLite;
- обязательные значения `web.env`;
- JSON и базовая структура `awg0.conf`.

Проверка выполняется сразу после создания и ещё раз до восстановления. Повреждённая копия блокируется до любых изменений. Старую проверяемую копию RC4 можно один раз перевести в формат RC5 с отдельным manifest.

Копия не является образом всей Ubuntu: системные пакеты, посторонние проекты и журналы в неё не включаются.

## SSH-меню

Из видимого меню полностью удалены старые дублирующие пункты 2, 5, 8 и 12. Оставшиеся действия перенумерованы без пропусков:

```text
 1. Состояние панели и адрес
 2. Перезапустить веб-панель
 3. Перезапустить Panel, AWG, Nginx и Agent
 4. Сменить пароль администратора
 5. Сбросить браузерные сессии и CSRF
 6. Проверить клиентов и подключения
 7. Проверить Cluster и SG-Node
 8. Проверить Cascade
 9. Создать резервную копию
10. Проверить резервную копию
11. Восстановить резервную копию
12. Переименовать сервер
13. Проверить и установить обновление
14. Полностью удалить SG-AWG-Panel
```

Аварийная команда `sudo sg-awg-panel repair-access` сохранена, но не занимает место в обычном меню.

## GitHub main, installer и uninstall

- отдельно скачанный `install.sh` получает полное дерево `main`, а не отсутствующий тег;
- updater также получает `main`;
- uninstall доступен напрямую из `main` и после неполной установки;
- installer заранее проверяет TCP `18080`, выбранный публичный TCP-порт и UDP `585`;
- один installer используется для Controller и будущей SG-Node;
- один updater используется на обеих EC2;
- публикационный CMD делает только commit/push в `main` и не создаёт Release или тег.
