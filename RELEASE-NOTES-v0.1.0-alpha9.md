# SG-AWG-Panel v0.1.0 Alpha 9

Alpha 9 завершает раздел **Server** и не меняет архитектуру HTTPS из Alpha 8.

## Server

- автоматическое определение публичного IPv4 и внешнего интерфейса сохранено;
- Endpoint, UDP-порт, клиентская сеть, DNS и MTU редактируются из одной формы;
- добавлены профили маскировки AWG 2.0:
  - **Стандартный** — проверенные текущие значения;
  - **Усиленный** — более интенсивная стартовая junk-последовательность;
  - **Пользовательский** — ручные Jc/Jmin/Jmax, S1–S4, H1–H4 и I1–I5;
- H1–H4 можно пересоздать одной кнопкой без смены серверных и клиентских ключей;
- форма заранее показывает последствия каждого изменения.

## Безопасное изменение сети

При смене клиентской сети:

- требуется отдельное подтверждение;
- перед применением создаётся резервная копия;
- все существующие клиенты автоматически получают последовательные адреса в новой сети;
- вместимость новой сети проверяется до записи;
- пользователь получает явное напоминание заново скачать `.conf` или QR;
- при ошибке запуска AWG база, адреса клиентов, `awg0.conf` и состояние службы возвращаются автоматически.

## UDP и NAT

- новый UDP-порт проверяется на занятость другой службой;
- рядом с полем показано напоминание об AWS Security Group или firewall;
- на странице Server отображаются:
  - состояние AWG;
  - автозапуск;
  - наличие `awg0`;
  - UDP listener;
  - IPv4 forwarding;
  - NAT MASQUERADE;
  - внешний интерфейс;
  - наличие рабочего `awg0.conf`.

## Совместимость

- обновление предназначено для `v0.1.0-alpha8`;
- база, ключи, клиенты, Access-ссылки, HTTPS, Nginx и сертификаты сохраняются;
- обычное обновление кода не перезапускает AWG до нажатия кнопки применения Server;
- один интерфейс `awg0`, IPv4, Ubuntu 22.04/24.04.

## Исправления проверки

- параметры формы проверяются до создания резервной копии и до записи в SQLite;
- полный набор тестов запускается в изолированном окружении;
- при ошибке публикации локальный `main` автоматически возвращается к чистой Alpha 8.

## Важно

Изменение Endpoint, UDP-порта, сети, DNS, MTU или параметров маскировки требует повторного скачивания клиентских конфигураций. «Усиленный» профиль создаёт больше стартового служебного трафика и не гарантирует лучшую работу в каждой сети.
