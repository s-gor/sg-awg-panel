01 · НАЧАЛО РАБОТЫ

Установка и первый вход

  1. 1
    Используйте чистую Ubuntu Server 24.04 LTS

    Панель устанавливает AmneziaWG, Nginx, Python runtime, SQLite и собственные systemd-службы.

  2. 2
    Запустите sudo bash install.sh

    Все вопросы мастер задаёт до применения основной конфигурации. Технический вывод сохраняется в журнал.

  3. 3
    Откройте адрес, показанный установщиком

    Войдите с паролем администратора. Первой открывается System → Resources.

  4. 4
    Проверьте AWG Server и создайте клиента

    После настройки сервера клиентская конфигурация доступна в Clients.

Перед установкой

Не устанавливайте панель поверх сервера с неизвестными правилами nftables, чужим WireGuard/AWG runtime или важным Nginx-сайтом без резервной копии.

02 · SYSTEM

Ресурсы, службы и журналы

Resources

Показывает CPU, память, диск и load average. Состояние памяти оценивается по доступной памяти, а файловый кэш не считается безвозвратной потерей RAM.

Logs & Diagnostics

Проверяет службы панели, AmneziaWG, Traffic Rules и Nginx. Здесь же скачивается безопасный диагностический отчёт без приватных ключей.

Что проверять после перезагрузки

Статус панели, наличие интерфейса AWG, UDP-порт сервера, IP forwarding, nftables и доступность backend только на loopback.

03 · CLIENTS

Создание и управление клиентами

  1. 1
    Нажмите «Добавить клиента»

    Укажите имя, комментарий и при необходимости срок действия.

  2. 2
    Скачайте .conf или используйте персональную ссылку

    Ссылка всегда отдаёт актуальную конфигурацию и может быть отдельно отключена или перевыпущена.

  3. 3
    Выберите выход

    По умолчанию клиент использует SG-AWG Gateway. Дополнительно доступны Block, отдельный Outbound или Cascade.

Удаление и перевыпуск

«Перевыпустить» меняет ключи клиента и делает старую конфигурацию недействительной. «Удалить» полностью убирает peer из серверной конфигурации.

04 · AWG SERVER

Сервер AmneziaWG

Раздел хранит параметры интерфейса awg0, публичный endpoint, UDP-порт, серверную сеть, MTU и параметры маскировки AmneziaWG.

Что можно менять безопасно

Публичный адрес, порт, MTU и профиль маскировки меняются через форму с проверкой. Панель создаёт резервную копию и применяет конфигурацию через управляемую службу.

Что произойдёт с клиентами

Изменение endpoint или порта требует скачать обновлённые клиентские конфигурации. Изменение приватного ключа сервера делает старые клиентские профили недействительными.

Не копируйте параметры наугад

Jc/Jmin/Jmax/S1–S4/H1–H4 должны оставаться согласованными на сервере и клиентах. Используйте профиль панели или экспертный JSON с обязательной проверкой.

05 · NETWORK

Traffic Rules и Outbounds

Traffic Rules

Правило выбирает клиентов, домены/IP, протокол и порты, затем направляет совпадение в Block или конкретный Outbound. Обычный SG-AWG Gateway является маршрутом по умолчанию и не требует отдельного правила.

Outbounds

Outbound — отдельная клиентская конфигурация AmneziaWG, через которую сервер выпускает выбранный трафик. Перед сохранением панель проверяет синтаксис, endpoint, адрес и full-tunnel AllowedIPs.

DNS

Доменная маршрутизация работает через управляемый DNS-path. Отключение DNS-контроля может сделать доменные правила неэффективными.

JSON

Current JSON — основной экспертный инструмент. «Проверить» не применяет изменения; «Сохранить и применить» доступно только для текста, прошедшего актуальную проверку.

06 · CASCADE

Клиент → Сервер подключения (Inbound) → Сервер выхода (Outbound) → Интернет

Сервер подключения (Inbound) — сервер, к которому подключается VPN-клиент. Сервер выхода в интернет (Outbound) — сервер, публичный IP которого видят сайты. В этом разделе Inbound и Outbound обозначают роли серверов в каскаде, а не отдельные пользовательские профили.

Вариант 1 — сервер из Cluster

  1. 1
    Подключите SG-Node

    В Cluster должны быть два готовых сервера: Controller и SG-Node либо две SG-Node.

  2. 2
    Выберите Inbound

    Это сервер, уже указанный в обычной конфигурации клиента.

  3. 3
    Выберите Outbound

    Его публичный IP будут видеть сайты и сервисы.

  4. 4
    Нажмите «Включить Cascade»

    Controller сам создаст служебный peer и применит серверный маршрут через Agent.

  5. 5
    Проверьте сервер и клиента

    Сначала подтвердите серверный маршрут. Затем откройте на VPN-клиенте любой сайт и выполните проверку подключения. Переподключать VPN не требуется.

Вариант 2 — другой самостоятельный сервер

  1. 1
    На сервере выхода (Outbound)

    Откройте Cascade → «Другой сервер» и нажмите «Создать ссылку для Cascade». Ссылка действует 30 минут; новая ссылка заменяет предыдущий служебный доступ.

  2. 2
    Скопируйте ссылку

    Она содержит только отдельный служебный AWG-профиль Cascade. Обычным пользователям её не передают.

  3. 3
    На сервере подключения (Inbound)

    Откройте тот же экран, вставьте ссылку и нажмите «Проверить ссылку и включить Cascade».

  4. 4
    Дождитесь проверки серверного маршрута

    Панель проверяет Outbound, policy routing и IP сервера выхода. Это ещё не подтверждение работы устройства.

  5. 5
    Проверьте VPN-клиент

    Откройте на клиенте любой сайт и нажмите «Проверить подключение клиента». Панель подтверждает свежий handshake, трафик и правильную метку маршрута Cascade; старые сессии при переключении сбрасываются автоматически.

Где создавать клиентов

Только в разделе Clients на сервере подключения (Inbound). Новые клиенты также автоматически используют активный Cascade.

Что означают статусы

Серверный маршрут готов означает, что Inbound связался с Outbound и применил маршрут. Cascade работает появляется только после проверки реального VPN-клиента. Если клиент не работает, нажмите «Повторить проверку», прочитайте простое сообщение об ошибке и откройте «Технические детали» только при необходимости.

Как отключить Cascade и начать заново

Для Cluster достаточно нажать «Отключить Cascade и начать заново»: Controller вернёт Inbound на прямой выход и удалит управляемый служебный маршрут.

Для двух самостоятельных панелей полный сброс состоит из двух отдельных шагов:

  1. 1
    На сервере подключения (Inbound)

    Нажмите «Отключить Cascade и начать заново». Панель удалит локальный маршрут Cascade и вернёт клиентов на прямой выход.

  2. 2
    На сервере выхода (Outbound)

    Откройте Cascade → «Другой сервер» и нажмите «Удалить служебный доступ Outbound». Только после этого служебный peer удалён на обеих сторонах.

После полного сброса можно создать новую ссылку и подключить другой Outbound. Клиентов пересоздавать не нужно.

Один сервер не может быть обеими сторонами

Inbound и Outbound должны быть разными. Для Cluster недоступные SG-Node заблокированы. Для самостоятельного сервера ссылка проверяется до применения.

07 · CLUSTER И SG-NODE

Четыре прямых шага

Один универсальный установщик, одно имя и одна команда подключения. Любая полная панель может быть Controller или SG-Node. Agent сам передаёт адреса и метрики; ручной ввод IP и скрытые действия не нужны.

  1. 1
    Установите вторую полную панель

    На обеих EC2 запустите один и тот же файл 0.7.0-RC3-INSTALL-SG-AWG-PANEL.run. Каждая панель остаётся самостоятельной, пока вы явно не подключите её к Controller.

  2. 2
    Введите имя и сразу получите команду

    Нажмите «Добавить и показать команду». Одноразовая команда с enrollment token появится прямо под формой и действует 30 минут.

  3. 3
    Выполните команду на SG-Node

    После сообщения Подключение завершено. Agent active вернитесь в Cluster. Статус «В сети», адрес, службы, heartbeat и реальный UDP-порт 585 появятся автоматически.

  4. 4
    Добавьте клиента

    Откройте Clients, нажмите «Добавить клиента» и выберите SG-Node как сервер подключения.

Повторное подключение

У ожидающей или недоступной SG-Node нажмите «Создать команду подключения» прямо в мастере Cluster. Новая запись не создаётся, старый agent token отзывается, а команда сразу появляется на этой же странице.

Защита от дублей

SG-Node с одинаковым именем второй раз не создаётся. Безопасные пустые дубли старых версий удаляются автоматически; подключённая SG-Node и её клиенты никогда не затрагиваются.

Почему SG-Node не в сети

Проверьте время, доступ Node к HTTPS-адресу Controller и журнал journalctl -u sg-awg-node-agent.service -n 100 --no-pager. Heartbeat идёт от Node к Controller, отдельный управляющий порт на Node не требуется.

08 · SECURITY

Доступ к панели и защита

Nginx и backend

Браузер обращается к публичному Nginx. Python backend слушает loopback и не должен быть доступен напрямую извне.

HTTPS

Укажите домен, направленный на сервер, и включите HTTPS. Certbot должен пройти HTTP-проверку. При лимите Let's Encrypt панель показывает точное время, после которого запрос можно повторить.

Allowlist

Ограничивает веб-интерфейс по IP/CIDR. Перед сохранением убедитесь, что текущий адрес входит в список, иначе можно потерять доступ.

Пароль и сессии

Смена пароля завершает активные сессии. Вкладка сессий позволяет отозвать отдельный браузер или все остальные входы.

Cloudflare

Cloudflare Proxy/Tunnel может скрыть прямой адрес панели, но не заменяет пароль, HTTPS, allowlist и обновления.

09 · MAINTENANCE

Резервные копии и обновления

Backup

Резервная копия включает базу данных, управляемые конфигурации и настройки панели. Перед восстановлением панель проверяет архив и создаёт страховочную копию текущего состояния.

Restore

После восстановления службы пересобираются и проверяются. При ошибке используется rollback.

Updates

Обновление загружает исходники, делает backup, запускает проверки и только затем переключает рабочую версию. Полный технический журнал сохраняется.

Cluster

Backup Controller хранит записи и токены SG-Node. Конфигурации и локальные журналы каждой Node нужно резервировать отдельно.

10 · ДИАГНОСТИКА

Что приложить к сообщению об ошибке

  • Версию SG-AWG-Panel и Ubuntu.
  • Название действия и точный текст ошибки.
  • Безопасный отчёт System → Logs & Diagnostics.
  • Для Cascade — состояние шага проверки, endpoint без приватного ключа и назначенных клиентов.
  • Для SG-Node — статус Agent, время последнего heartbeat и результат задания Diagnostics.
Не публикуйте

PrivateKey, PresharedKey, access token, enrollment token, agent token, полный backup и содержимое клиентского .conf.

11 · FAQ

Частые вопросы

Нужен ли домен?

Для обычного HTTP по IP — нет. Для нормального публичного HTTPS нужен домен, направленный на сервер. Для Agent желательно использовать стабильный HTTPS-адрес Controller.

Нужен ли GitHub при установке из архива?

Нет. Исходники панели устанавливаются из загруженного архива. Интернет нужен для системных пакетов, AmneziaWG и других зависимостей.

Чем Cascade отличается от Cluster?

Cascade — маршрут трафика через два сервера. Cluster — централизованное наблюдение и управление несколькими SG-Node. Их можно использовать отдельно или вместе.

Можно ли получить российский IP через Cascade?

Да, если Exit-сервер физически находится в России и его провайдер выдаёт российский IP. Сам Cascade страну IP не создаёт.

Нужно ли менять конфигурации клиентов после включения Cascade?

Нет. Клиент по-прежнему подключается к своему серверу подключения; маршрут до сервера выхода применяется на стороне серверов.
