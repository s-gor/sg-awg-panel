# Резервные копии и обновление

## Расписание резервных копий

Откройте **Backups** и выберите:

- каждый час;
- каждые 6 часов;
- ежедневно;
- раз в неделю;
- выключено.

Можно хранить от 1 до 365 последних копий. Таймер systemd использует `Persistent=true`, поэтому пропущенная копия создаётся после включения сервера.

Проверка:

```bash
systemctl status sg-awg-backup.timer --no-pager
systemctl list-timers sg-awg-backup.timer
```

## Что входит в копию

- SQLite-база;
- рабочий `awg0.conf`;
- metadata с причиной и состоянием службы.

Перед изменением Server, клиентов, Routing и восстановлением создаётся страховочная копия.

## Проверка обновлений

Откройте **Settings → Обновления** и нажмите **«Проверить сейчас»**. Панель читает теги репозитория GitHub и сравнивает их с установленной версией.

## Фоновое обновление

Кнопка обновления запускает отдельную systemd-задачу. Web-процесс не выполняет обновление внутри HTTP-запроса.

Перед изменениями сохраняются:

- каталог проекта;
- SQLite-база;
- `web.env`;
- конфигурация Nginx;
- systemd units панели, backup timer и recovery.

После установки выполняются:

1. миграция базы;
2. проверка backend `/health`;
3. `nginx -t`;
4. проверка systemd-служб.

Если этап завершается ошибкой, updater останавливает новую панель и автоматически восстанавливает предыдущие файлы, базу, env, Nginx и units.

Журнал:

```text
/var/lib/sg-awg-panel/update.log
```

Статус:

```text
/var/lib/sg-awg-panel/update-status.json
```

## Ручное обновление

```bash
sudo env SG_AWG_PANEL_VERSION=v0.1.0-alpha7 \
  bash /opt/sg-awg-panel/deploy/update-from-github.sh
```

Обычное обновление панели не перезапускает рабочий AWG-туннель.
